Ten things I didn't know about HTML

- Mathias Bynens - Twitter: @mathias - Video: http://vimeo.com/52851509

Mathias showed us some funny things he found out using HTML. For example he tried to use a name for setting a background color, for example: 

<div bgcolor="fronteersconf"> Fronteers Conf </div>

returns the Fronteers yellow logo color. Or "sick" returns: #000c00. Try it for yourself to find out what your color is. It also gives different results when you remove the spaces in your name: bgcolor="your name".

Mathias also advises not to be obsessive about HTML validation. It is build by people and also has bugs. But most of all, functionality is more important than validation.

“Validation is a tool, not a goal.” - Mathias Bynens

Characters don't need to be output escaped. The browser will render it OK, It is also DTD valid.

He also found out that "font family" does not need to be quote. It's rendered correctly: 

font-family: Comic Sans MS;

But a font name beginning with a number does need quotes.

Beyond simple transitions, with a pinch of JavaScript

- Peter Nederlof - @peterlof - Video: http://vimeo.com/53317255

To create animations we had to rely on custom javascript tools and libraries. We now can accomplish this with a simple CSS transition. However transitions still have their limits. Peter shows how to step beyond some of these limits.

Benefits of CSS transitions

  • complete animation engine
  • no per-frame script calcuations
  • optimized in every way!

You should also use 3d transforms. It has no reflows. It is hardware accelerated, also on mobiles. It is battery friendly and widely supported. Except for IE, of course.

But there is a problem: transitions have no stop. You can't stop or interrupt an transition. Querying the CSS property values might not always work.

Things to keep in mind:

  • The transitionEnd event is not fired when a transition is interrupted.
  • Assigning styles via the javascript Api has a very high specifity.
  • Make fallbacks for critical cases: like left/top positioning for <IE8, instead of using transformations.
  • Use progressive enhancement.
  • And most important, convince your client or project manager that this is OK.

JS Minty Fresh: Identifying and Eliminating Smells in Your Code Base

- Rebecca Murphey - Twitter: @rmurphey - Video: http://vimeo.com/53416986

Rebecca talks about javascript code smell and how to solve this. It was a great talk with nice examples. I didn't even know that you can create html "template" and call it in javascript to make it more clean. That was a nice surpise. The video can tell you a lot more than these notes.

Code smells are a suggestion of a deeper problem

The harm is that debugging is harder, maintainability suffers and features are hard to add. You never come back to fix it later. So create clean code from the start.

The smell: Repetitive code
The fix: A function, object

The smell: complex functions
The fix: More & smaller functions:keep functions short and simple and avoid complex if/else logic

The smell: async acrobatics
The fix: deferrers:manage and leverage asynchronicity

The smell: html in javascript
The fix: place it in the template. You can use _.template function from Underscore.js.

You can download the presentation from: Rebecca Murpheys website

What the legacy web is keeping from us

- Alex Russell - Video: http://vimeo.com/53373706

Progress is a man-made process, and like most man-made processes progress can be sped up and slowed down. Progress is good for people, for us. However it requires change and how many changes should be made and what changes should be made.

“Accessibility is the most important thing! In the computing world.” - Alex Russell

Progress delayed is progress denied! For example the canvas tag or rounded cornes, svg. Some took 6-8 years, while it was proposed in the beginning of 2000. Optimistically, new features shipped today will not be usable until 2016!

Users do what they are asked to do. We can change that users start using new browsers.
